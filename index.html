Introduction

Ant is a PHP framework for rapid web application development, designed to be lightweight and to have a consistently intuitive MVC-based structure that is as easily navigable in the directory structure as it is the code.

Features and Goals

Organisation

  • Ability to scale from tiny projects to hundreds of files.
  • Keep PHP files small by relying heavily on lazy-loading.
  • Separation of PHP and HTML using optimized string replacement
  • Separation of database queries, to allow for reusability, extensibility.
  • Flexible routing in XML files
  • Accessing of controllers directly via whitelisted http requests.
  • "Channelling" option, to change output based on the same route.

Tools

  • Handling of JavaScript and Less.css compiling, optimization.
  • Included basic database structure for the everyday application
  • Basic handling of user functionality, like login
  • Basic authentication class, using Facebook/Google API
  • Internalization of languages.
  • Client detection (e.g. mobile) and custom client views.

Requirements / Dependencies

  • PHP 5.3+/MySQL on Apache
  • Underscore.php

Setup

Download the zip file or fetch the repository from Github.

Create a new mysql database, using the name of your project, and then import ant.sql to it. After import you can change the prefix of the tables to the name of your project or just leave it as "ant_".

After creating the database and importing the basic Ant structure, you'll need to confirm the local/remote mysql configuration inside [version]/config/mysql.php. Within this file you can set your database name, as well as your table prefix.

Open the root to your project in your browser, and you should see the Ant Logo.

Confirming setup by testing

To make sure Ant is working properly, you can run the included unit tests, using PHPUnit. You can try run these tests immediately, using the url /test.php. This will attempt to run the tests via command line using the exec function in PHP. If this doesn't work, or you don't have PHPUnit installed, consult the Unit Tests section below.

Basic Concepts

The Directory Structure

Using Ant as your framework requires an understanding of the directory structure. Consider the essentials of the root directory.

0.1.0/
lib/
tests/
.htaccess

In the root, you'll notice a directory with a version number. This is where the current iteration of your application is stored. You can store as many sibling versions as you want here, the purpose being that you can quickly rollback to a previous version if needs be, by changing one line in the .htaccess file.

Note, lib/ is used to store external libraries, and tests/ are the unit tests of the application. Let's burrow down into the application folder:

0.1.0/
0.1.0/
  - app/
  - config/
  - lib/
  - public/

The /app/ folder stores all the aspects of your application, like models, views, and controllers.

The /config/ folder stores your application's configuration in a variety of contexts.

The /lib/ folder contains portable code, designed for your application but potentially standalone.

The /public/ folder contains resources that will be accessible via http, like JavaScript, Css styles, and images. Furthermore, this folder contains xml that maps the templates to application data from the view.

0.1.0/app/
0.1.0/
  - app/
    - cache/
    - classes/
    - modules/

Inside the app directory is a /cache/ directory used for any kind of server-side caching that needs to take place.

The /classes/ directory hosts the core application logic, as well as contextual classes (the models), used for specific contexts (e.g. User or Products).

The /modules/ directory is where dynamic logic exists, based on the application's current state. Most of the files contained herein are lazy-loaded reusable functions.

0.1.0/app/modules/
0.1.0/
  - app/
    - modules/
      - context/
        - controllers/
        - queries/
        - views/
      - shared/
        - views/

The /modules/ directory is divided into two concepts, Shared and Context. You'll see this division elsewhere in Ant. Simply, this explains resources that are always available (like a header and footer), and resources that depend on the current context.

Currently /views/ is the only directory in the Shared space. This is because controllers and queries usually apply to a context.

0.1.0/app/modules/shared/views/

Contained inside the /views/ directory, either in the shared or context spaces, are the Clients. This is another dimension of Ant used throughout, which allows the application to create different output depending on the current client (e.g. mobile).

Naturally, you can import the view of one client to another client's view, since everything in Ant is namespaced and won't conflict.

The main client is /web/. This is the client for the desktop browser.

0.1.0/public/

Let's have a quick look at the public directory. This is where your public resources like JavaScript and Css and images will be stored. The public folder also uses the three dimensions already discussed. The Shared space, the Context space, and the Client space.

Note, .htaccess automatically rewrites any requests from public/ to 0.1.0/public/ so you don't need to worry about versioning your resources inside the html.

0.1.0/
  - public/
    - clients/
      - web/
        - css/
        - javascript/
        - templates/
          - en_gb/

Here you can see that the resources are arranged into different /clients/, but if you needed to, you could reference resources from other clients, within your html. You would load a resource like a stylesheet as follows:

<link type="text/css" href="public/../reset.css" />

Note that Ant can dynamically load generated stylesheets using less.css, as well as organize them into several defined files according to a "namespace". Directly loading JavaScript or Css resources should be avoided for performance and caching reasons.
